项目整体定位

这是一个本地老虎机算法与机台协议集成项目，核心目标是：
1) 提供多游戏实例（当前内置 3 款：1700、3999、3998）的开奖结果生成
2) 通过 RTP 分档、分池与概率闸门控制，稳定长期回报率
3) 兼容本地彩金与联网彩金两套路径
4) 通过 idea 协议命令（JSON cmd）对外提供初始化、下注、转轮结果、配置读写等能力

从代码结构看，这个项目是“算法层 + 业务控制层 + 协议适配层”的典型分层架构，工程目标偏向“机台实战可控性”和“调参与调试可观测性”。


目录与模块分层分析

一、启动与主循环层
- 入口在 application/IDEA/app/source/main.c
- main 流程：
  - qs_init()、app_init()、qs_start()
  - _LocalDebug 下执行压力测试、RTP 统计输出
  - 最终进入 while(1) { qs_exec(); }

说明：主循环由 qs 框架驱动，算法通过事件/命令触发执行。


二、协议适配层（外部交互入口）
- 核心在 application/IDEA/app/source/idea_cmd.c
- 通过 SenvReadCallback 对 cmd 分发，典型命令包括：
  - 20000 初始化游戏
  - 20002/20003 读写配置
  - 20101 请求一局开奖结果（spin 核心入口）
  - 20200 切换游戏
  - 20030 联网彩金入账

特点：
- 对外是“命令驱动 + int 数组协议”，内部再转算法对象
- 协议层同时做账户分数更新（player_bets_info_update）与结果打包
- 当前仍有多个 TODO（协议行为未完全闭环）


三、算法与控制中枢层（最关键）

1) Dll.c：一局结果编排总控
文件：application/IDEA/app/source/Control/Dll.c

职责：
- 游戏实例获取/切换（get_instance、DLL_GameSwitch）
- 候选局生成（普通/免费/Bonus）
- 交给 TableControl 做放行判定（池子、区间、概率）
- 拒绝时强制输局，放行时回填 OutResult_t
- 统计 DebugInfo（概率、RTP 分项、次数口径）

本质是“候选结果生成器 + RTP 风控闸门执行器 + 输出适配器”。

2) TableControl.c：RTP 闭环控制器
文件：application/IDEA/app/source/Control/TableControl.c

实现能力：
- 4 个主要池 + 1 个联网预算池
  - gBasePool、gFreePool、gBonusPool、gJackpotPool、gNetJackpotPool
- 每局下注按 profile 注入池子（不是命中才注水）
- 候选支付需经过三层约束：
  - 区间校验（min/max 倍数）
  - 软池校验（允许有限负债）
  - 概率门校验（支持 override）
- 拒绝统计可细分到具体原因（range/pool/pass）

这是工程价值最高的模块，具备“可调、可观测、可控”的生产特征。


四、游戏实例与配置层

1) GameManager + GameRegistry
- GameManager 负责实例生命周期（注册、切换、注销、当前实例）
- GameRegistry_InitAndRegisterDefaults 注册三款游戏固定配置（线数、符号、roll 表 id 等）

优点：实例管理集中，切换清晰。
注意：GAME_INSTANCE_ID_MAX 当前为 3，扩容需要改常量及关联数组逻辑。

2) RtpConfig
文件：application/IDEA/app/source/Control/RtpConfig.h

静态表维护：
- 国内/海外双区域
- 每区 5 档 RTP
- 每档分项占比、概率门、倍数区间

优点：参数集中，方便运营调档。
风险：静态写死，若要在线热更，需增加外部配置加载机制。


五、奖池层

1) 本地彩金：LotteryManager + Lottery
- LotteryManager_OnPlay 根据下注档位注入
- LotteryManager_TryGetLottery 先产生“候选彩金”
- 真正落账在 LotteryManager_CommitLottery（仅主结果放行后提交）

这个“两阶段提交”设计是正确的，避免“候选命中但主结果拒绝”导致账务不一致。

2) 联网彩金
- 协议命令 20030 入账
- 同步调用 DLL_OnJackpotOnlineWin 进入 TableControl 统计和预算池扣减

说明：联网彩金在项目中是“外部下发 + 本地统计预算”模式。


核心业务流程（从下注到开奖）

1. 协议层收到 20101
2. 读取玩家下注，判定是否免费局
3. 普通局先尝试本地彩金候选
4. 调用 DLL_GetGameResultById
5. 内部生成候选矩阵/免费/Bonus 数据
6. TableControl_GetShotResult 判定是否放行
7. 拒绝则改输局；放行则写回 OutResult_t
8. 放行后如有彩金候选，则 CommitLottery
9. 返回协议数据，更新玩家账务与统计

这条链路逻辑完整，且“候选 -> 风控 -> 落地”边界清晰。


技术亮点与优点

1) RTP 控制模型成熟：分项池 + 区间 + 概率三重闸门
2) 统计可观测性好：TableControlStats 与 DebugInfo 口径较完整
3) 多游戏实例抽象明确：GameManager / GameRegistry 分责清晰
4) 协议接入简单直接：cmd 映射虽朴素但维护成本低
5) 调试能力强：_LocalDebug、_DebugControlMode 支持压测和定向结果调试


当前主要风险与技术债

1) 工作区有大量构建产物与 .vs 临时文件混入，污染版本管理与评审
2) 路径风格混用（/ 与 \）存在重复/镜像文件风险
3) idea_cmd.c 过长且分支密集，协议分发与业务逻辑耦合
4) 全局状态较多（如 g_CurrentGameInstance、全局池变量），并发化风险高
5) 调试宏分支较多，Debug/Release 行为差异风险较大
6) 部分接口容错依赖上层输入正确，边界保护可继续加强


性能与可维护性评估

- 性能：当前 C 实现和静态查表足够快，单机吞吐不是瓶颈
- 可维护性：中等偏低（主要受 idea_cmd.c、Dll.c 超长函数影响）
- 可扩展性：中等（新增 gameId 可行，但要改静态表和多处分支）
- 稳定性：中等偏高（风控闸门合理，但异常输入保护需要补强）


建议的演进路线（按优先级）

P0：
1) 补全 .gitignore（.vs、build、x64/Debug、中间产物）
2) 拆分 idea_cmd.c（配置/账务/开奖/彩金分域）

P1：
1) 将 GameRegistry 的游戏配置改为外部配置驱动（JSON 或二进制表）
2) 统一路径与编码规范，清理重复镜像文件
3) 补充回归测试脚本（重点验证 RTP 分项、拒绝原因、账务一致性）

P2：
1) 为 TableControl 增加运行时快照导出（便于线上观测与调参）
2) 整理宏开关策略，形成 Debug/Release 行为矩阵


结论

这是一个已经具备生产算法控制雏形的机台项目。
最有价值的部分是 TableControl 的 RTP 闭环与 Dll.c 的结果落地链路。

当前最大问题不在算法思路，而在工程治理层（构建产物污染、协议层耦合、文件组织复杂度）。
如果先完成工程边界治理与配置治理，项目可维护性和迭代效率会明显提升。
